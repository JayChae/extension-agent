## 0. 무엇을 왜 재나 (한 줄)

이 시스템 전체는 **"scourt 화면에서 버튼·입력칸을 정확히 집어내기"** 에 달려 있다.
content script 는 **isolated world** 라 페이지의 jQuery·이벤트리스너를 못 본다 →
**간접 신호**(태그·ARIA·cursor:pointer·tabindex·onclick속성·ID키워드)로 근사한다.
**이 간접 신호가 scourt(W2UI/eXBuilder6)에서 충분한가?** 이걸 4화면에서 잰다.

측정은 한 번에 둘을 한다:
- **C (휴리스틱):** isolated-world 안전 신호로 우리가 잡은 집합 = 실제 실력.
- **O (정답지/oracle):** 페이지 jQuery 가 알려주는, 진짜 클릭 핸들러가 붙은 집합 = 채점 답안.

핵심 산출물은 퍼센트 한 줄이 아니라 **(a) 🔴 MISSED 집합의 정체 + (b) 바인딩 방식
(per-element vs delegated)** 이다. 이게 "얇은 isolated content script" 전제의 생존을 결정한다.

## 1. 사전 등록 — "핵심 컨트롤" 정의와 세는 규칙 (변경 금지)

**핵심 컨트롤(core control):** 해당 화면에서 작업을 해피패스로 끝내려면 사람이 **한 번 이상
클릭·입력·선택해야 하는, 보이고 동작 가능한** 요소.

세는 규칙(엣지케이스 미리 확정):
1. **포착률은 "요소 개수"가 아니라 "컨트롤 종류(class)"로 센다.** 같은 '열람' 버튼이 표에
   20행 있어도 **1종**. (한 행을 잡으면 동일 구조 전 행을 잡으므로.) 렌더된 인스턴스 수는 따로 기록.
2. **표의 행 반복 버튼** = 1종. 가상 스크롤로 일부 행만 렌더되면 그 사실을 기록.
3. **페이징(다음/이전/페이지번호)** = 1종(번호마다 세지 않음). 단 작업상 페이지 이동이 필요할 때만 핵심.
4. **자동완성 위젯(`_acp_`)** = 입력칸 1종. 타이핑 후 뜨는 옵션 목록은 *조건부 집합*으로 별도 기록(핵심 종수엔 미포함, 단 "옵션을 잡았나"는 메모).
5. **커스텀 라디오/체크(`_rad_..._input_n`, `_chk_`)** = 사람이 고를 수 있는 *구분되는 선택지* 각 1종.
6. **숨김/비활성:** `display:none`·`visibility:hidden`·크기 0·`disabled`는 핵심에서 제외. 단
   **입력 후 활성화되는 폼 버튼**은 *활성 상태로* 측정한다(그게 진짜 사용 상태). 숨김/비활성 개수는 별도 기록.
7. **로그인/인증서(④):** 인증서 *선택* 컨트롤·HDD·확인 버튼은 핵심. **PIN 입력칸은 `fill_credential`
   대상이라 세긴 하되 값은 절대 읽지 않음**(도구도 비밀값 마스킹).
8. **편향 줄이기:** 채점자는 "내가 실제로 누르는 컨트롤" 목록을 *먼저* 적은 뒤 도구의 버킷 결과를 본다.

## 2. 버킷 정의 (도구가 자동 분류)

| 버킷 | 의미 | 함의 |
|---|---|---|
| `caught-native` | 휴리스틱 O + 네이티브 인터랙티브 태그 | 항상 정상 |
| `caught-aria` | 휴리스틱 O + ARIA role/state, 핸들러 확인됨 | 설계대로 작동 |
| `caught-keyword` | 휴리스틱 O + ID/class 키워드, 핸들러 확인됨 | 작동(사이트별 명명규칙 의존) |
| `caught-pointer` | 휴리스틱 O + cursor:pointer, 핸들러 확인됨 | 작동하나 취약(오탐 위험) |
| `caught-other` | 휴리스틱 O + tabindex/onclick, 핸들러 확인됨 | 보조 신호 |
| `FPcandidate-*` | 휴리스틱이 잡았지만 **핸들러 없음 & 네이티브 아님** | **오탐 후보(잡음).** 정밀도 위험 |
| **`MISSED` 🔴** | 휴리스틱이 **못 잡음**인데 **핸들러 있음** | **놓친 진짜 버튼. 핵심 위험 지표.** |

> 포착률(recall) ≈ (핵심 중 잡힌 종수) / (핵심 종수).
> 정밀도(precision) 위험 ≈ `FPcandidate-*` 개수(많을수록 모델이 잡음에 시달림).

## 3. ✅ 게이트 판정 규칙 (사전 확정)

- 판정 기준 = **4화면 가중 포착률(핵심 컨트롤 *종* 기준)**.
- **≥ 90% + `FPcandidate` 관리가능 + 🔴 집합이 학습 가능한 ID 패턴 → Phase 1 진행.**
  (이때 architecture.md §3 에 eXBuilder6 ID-접두사 키워드 신호를 정식 추가.)
- **70 ~ 90% → 조건부 진행.** 🔴 집합의 ID/class 접두사를 캐서 키워드 신호에 넣었을 때
  *서류상* 갭이 메워지면 진행, 아니면 ≤70 으로 취급.
- **⛔ ≤ 70%, 또는 바인딩이 delegated 우세라 요소별 탐지가 불안정, 또는 🔴 집합에 안정적
  시그니처가 없음 → 멈춤.** §3 지각 재설계(유력 후보: MAIN-world 지각 oracle 을 §4 의
  alert/confirm 훅에 이은 *두 번째* 허용 예외로 승격 / 최후수단 CDP 지각).

> 부산물(게이트 아님): 화면당 `index build ms` → Phase 3 지연 예산 기준값.
